# 도입사례 정리 (Case Study → data.csv)

사용자가 **회의록 원문** 또는 고객사 도입 관련 정보를 보내면, 아래 규칙대로 핵심 정보를 추출해 워크스페이스의 `data.csv`에 **한 행만 추가(append)** 한다.

## 스프레드시트 링크 (Slack 보고 시 필수)
- `data.csv`는 아래 구글 시트와 동기화된다. **작업을 끝내고 슬랙에 보고할 때마다 이 링크를 반드시 그대로 포함**한다. 단 한 번도 빠뜨리지 말 것.
- 링크(고정, 절대 변형 금지): https://docs.google.com/spreadsheets/d/1SHYSGggEpmoreAZNmXdiuuJQTmTiqGT_YauiRfvRRIo/edit
- 슬랙에서는 위 URL을 **한 줄에 단독으로** 붙여넣어 미리보기가 펼쳐지게 한다. `<...>`나 마크다운 `[](...)`로 감싸지 말고 raw URL 그대로 출력한다.

## 출력 파일
- 대상: 워크스페이스 루트의 `data.csv` (현재 작업 폴더 기준 `./data.csv`)
- 컬럼 순서(고정): `산업군,플랜,기업명,부서,도입 사유 / 후킹 포인트 / 활용 방법`
- 기존 내용(헤더·기존 데이터 행)은 **절대 수정·삭제하지 않고**, 파일 맨 끝에 새 행만 추가한다.
- 파일 2번째 줄(산업군·플랜 값들이 나열된 줄)은 "허용값 안내용 범례"이므로 데이터로 취급하지 말고 그대로 둔다.

## 추출 규칙
1. **산업군** — 아래 목록에서만 선택. 해당하면 여러 개도 가능(쉼표로 구분).
   - 허용값: 건기식, 금융, 의료/건강/보건, 푸드, 커머스/플랫폼, IT, 앤서비스, R&D/연구/리서치, 엔터테인먼트/미디어/예술, 생활/소비재, 주택/건설/부동산, 제조
   - 목록에 없는 산업군을 임의로 만들지 말 것. 애매하면 빈칸으로 두고 슬랙에서 사용자에게 한 번 확인한다.
2. **플랜** — basic, standard, professional, advance 중 하나. 회의록에 근거가 없으면 빈칸.
3. **기업명** — 고객사명.
4. **부서** — 담당 부서/팀. 없으면 빈칸.
5. **도입 사유 / 후킹 포인트 / 활용 방법** — 도입 이유, 관심을 끈 포인트, 실제 활용 방법을 1~3문장으로 요약. 회의록에 실제로 나온 내용만 쓰고 추측·창작은 금지.

## CSV 작성 형식
- 쉼표·줄바꿈·따옴표가 들어가는 값은 반드시 큰따옴표로 감싼다. 예: `"의료/건강/보건, 생활/소비재"`
- 산업군이 여러 개면 한 셀 안에서 쉼표로 구분하고 셀 전체를 큰따옴표로 감싼다.
- 인코딩은 UTF-8 유지. 헤더 줄은 다시 쓰지 않는다.

## 작업 절차
1. 받은 내용에서 위 5개 항목을 추출한다.
2. 추출 결과를 슬랙에 표 형태로 먼저 보여주고, `data.csv`에 추가하겠다고 알린다.
3. `data.csv`의 마지막 글자가 줄바꿈인지 먼저 확인한다. 줄바꿈으로 끝나지 않으면 줄바꿈을 한 번 넣어 마지막 행과 새 행이 붙지 않게 한다.
4. `./data.csv` 맨 끝에 새 행 1줄을 추가한다(기존 행은 그대로 보존).
5. 추가된 행 내용과 현재 총 데이터 행 수를 슬랙에 보고한다. **보고 메시지 맨 끝에는 항상 위 "스프레드시트 링크"를 한 줄에 단독으로 붙인다** (예: `시트에서 확인:` 다음 줄에 https://docs.google.com/spreadsheets/d/1SHYSGggEpmoreAZNmXdiuuJQTmTiqGT_YauiRfvRRIo/edit ). 링크 없이 보고를 끝내지 않는다.

## 안전 규칙
- data.csv를 덮어쓰거나 기존 행을 삭제하지 않는다. 오직 끝에 추가만 한다.
- 같은 기업명이 이미 data.csv에 있으면, 새로 추가하기 전에 "기존 행 수정 vs 새 행 추가" 중 무엇을 원하는지 사용자에게 묻는다.
- 근거가 부족한 항목은 비워 두고, 칸을 채우려고 사실을 지어내지 않는다.
- 번역(일본어·영어)은 이 스킬 범위 밖이다. 지금은 한국어로만 정리한다.
